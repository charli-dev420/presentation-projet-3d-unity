# Decision Pack / Dossier de decision

[EN](#english) | [FR](#francais)

## English

### Recommended Priority Demo

1. Dataset ReviewEval: small public-safe batch, keep/fix/reject decisions, readable export.
2. Splat Face: depth-card or 2.5D asset route using a shareable source.
3. Unity: import or handoff in a disposable project with pivot, scale, bounds, and material expectations visible.
4. CodexUnity / LocalAssetFactory: dry-run first, then controlled run only if the environment and assets are cleared.
5. Mob'ia / ccomf-unity: one product walkthrough with profile, job, artifact, review, and client handoff.

### For A Funder

- Fund live Unity, Blender, and ComfyUI-style validations.
- Fund a library of public or NDA-cleared use cases.
- Fund asset QA: bounds, triangle budget, textures, materials, profiles, and mobile constraints.
- Fund packaging of the partner evaluation kit.

### For A Unity Partner

- Bring project constraints and performance criteria.
- Pick the target format: prefab, scene element, depth-card, GLB, or package.
- Require a proof that can be imported or reviewed in a disposable Unity project.

### Decision Questions

1. Which asset output matters first?
2. Which live proof should be closed first?
3. Which data and visuals can be public, private, or NDA-only?
4. Which demo format builds the most confidence: written report, video, public repo, or Unity scene?
5. What single acceptance criterion would make the next step obvious?

## Francais

### Demo prioritaire recommandee

1. Dataset ReviewEval: petit lot public-safe, decisions garder/corriger/rejeter, export lisible.
2. Splat Face: route depth-card ou asset 2.5D avec source partageable.
3. Unity: import ou handoff dans projet jetable avec pivot, echelle, bounds et attentes materiaux visibles.
4. CodexUnity / LocalAssetFactory: dry-run d'abord, puis run controle seulement si environnement et assets sont autorises.
5. Mob'ia / ccomf-unity: walkthrough produit avec profil, job, artefact, revue et handoff client.

### Pour un financeur

- Financer les validations live Unity, Blender et type ComfyUI.
- Financer une bibliotheque de cas d'usage publics ou sous NDA.
- Financer la QA asset: bounds, budget triangles, textures, materiaux, profils et contraintes mobile.
- Financer le packaging du kit d'evaluation partenaire.

### Pour un partenaire Unity

- Apporter contraintes projet et criteres de performance.
- Choisir le format cible: prefab, element scene, depth-card, GLB ou package.
- Exiger une preuve importable ou revisable dans un projet Unity jetable.

### Questions de decision

1. Quelle sortie asset compte d'abord?
2. Quelle preuve live doit etre fermee en premier?
3. Quelles donnees et visuels peuvent etre publics, prives ou NDA-only?
4. Quel format de demo rassure le plus: rapport ecrit, video, repo public ou scene Unity?
5. Quel critere d'acceptation rendrait la prochaine etape evidente?
