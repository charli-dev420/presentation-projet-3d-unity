# Demo Scenarios / Scenarios de demonstration

[EN](#english) | [FR](#francais)

## English

### Demo Rule

Every demo should start with the decision it needs to produce. A good session answers one question cleanly: should this source be used, should this asset candidate move forward, is the Unity import acceptable, or is the product flow worth a deeper review?

### 1. Dataset Review With Dataset ReviewEval

Audience: buyer, dataset lead, technical artist.

Agenda: open a cleared sample set, compare image readability, framing, duplication risk, target use, and provenance. Mark each source keep/fix/reject and export a JSON/CSV/Markdown summary.

Success criteria: another reviewer can understand why the batch was filtered without opening the original working folder.

Public artifact: redacted export plus short decision notes.

### 2. Splat Face / Splat Facade Baker Mobile Candidate

Audience: Unity/mobile studio, technical artist.

Agenda: choose one facade or character-facing source, describe the expected 2.5D or depth-card route, define mobile constraints, and prepare the acceptance criteria before any visual polish.

Success criteria: the team knows what must be checked before the candidate is useful in Unity: silhouette, depth behavior, texture readability, approximate weight, scale, and scene role.

Public artifact: pipeline card, QA checklist, and written asset criteria.

### 3. CodexUnity / CodexToUnity Dry-Run

Audience: technical collaborator, tool reviewer.

Agenda: show the job shape, manifest fields, sockets/import handoff, normalization expectation, and dry-run trace without relying on a polished generated asset.

Success criteria: the bridge role is clear enough to choose the next live validation: which input, which manifest, which Unity check, and which result file.

Public artifact: written trace, architecture note, and import checklist.

### 4. Mob'ia / ccomf-unity Product Walkthrough

Audience: product partner, buyer, funder.

Agenda: walk through one profile, one job, one artifact, one review state, and one Unity/web/mobile client path. Keep the conversation on the operator decision: what is created, what is reviewed, and what the next action is.

Success criteria: the product can be discussed as a real workflow instead of a generic AI dashboard.

Public artifact: user journey, integration points, and scoped demo plan.

### 5. LocalAssetFactory Unity Handoff

Audience: Unity reviewer, partner, technical artist.

Agenda: use a public or cleared asset, run preflight criteria, normalize the candidate, pass the manifest to Unity, inspect the import, and document accept/fix/reject.

Success criteria: the asset decision is grounded in concrete Unity observations: scale, orientation, material assignment, naming, scene fit, and mobile readiness.

Public artifact: QA result summary or scoped evidence pack.

## Francais

### Regle Demo

Chaque demo doit commencer par la decision a produire. Une bonne session repond proprement a une question: faut-il utiliser cette source, faut-il avancer ce candidat asset, l'import Unity est-il acceptable ou le flow produit merite-t-il une revue plus profonde ?

### 1. Revue Dataset Avec Dataset ReviewEval

Public cible: acheteur, dataset lead, artiste technique.

Deroule: ouvrir un lot d'exemples autorises, comparer lisibilite image, cadrage, risque de doublon, usage cible et provenance. Marquer chaque source garder/corriger/rejeter et exporter un resume JSON/CSV/Markdown.

Criteres de succes: un autre reviewer peut comprendre pourquoi le lot a ete filtre sans ouvrir le dossier de travail original.

Artefact public: export redige plus notes courtes de decision.

### 2. Candidat Mobile Splat Face / Splat Facade Baker

Public cible: studio Unity/mobile, artiste technique.

Deroule: choisir une source facade ou orientee personnage, decrire la route 2.5D ou depth-card attendue, definir les contraintes mobile et preparer les criteres d'acceptation avant le polish visuel.

Criteres de succes: l'equipe sait ce qui doit etre controle avant que le candidat soit utile dans Unity: silhouette, comportement profondeur, lisibilite texture, poids approximatif, echelle et role en scene.

Artefact public: carte pipeline, checklist QA et criteres asset rediges.

### 3. Dry-Run CodexUnity / CodexToUnity

Public cible: collaborateur technique, reviewer outil.

Deroule: montrer la forme du job, les champs de manifest, le handoff sockets/import, l'attente de normalisation et la trace dry-run sans dependre d'un asset genere poli.

Criteres de succes: le role du pont est assez clair pour choisir la validation live suivante: quelle entree, quel manifest, quel controle Unity et quel fichier resultat.

Artefact public: trace redigee, note architecture et checklist import.

### 4. Walkthrough Produit Mob'ia / ccomf-unity

Public cible: partenaire produit, acheteur, financeur.

Deroule: parcourir un profil, un job, un artefact, un etat de revue et un chemin client Unity/web/mobile. Garder la discussion sur la decision operateur: ce qui est cree, ce qui est revu et quelle action vient ensuite.

Criteres de succes: le produit peut etre discute comme un vrai workflow, pas comme un dashboard IA generique.

Artefact public: parcours utilisateur, points d'integration et plan demo cible.

### 5. Handoff Unity LocalAssetFactory

Public cible: reviewer Unity, partenaire, artiste technique.

Deroule: utiliser un asset public ou autorise, lancer les criteres preflight, normaliser le candidat, passer le manifest a Unity, inspecter l'import et documenter accepter/corriger/rejeter.

Criteres de succes: la decision asset repose sur des observations Unity concretes: echelle, orientation, assignation materiaux, nommage, integration scene et readiness mobile.

Artefact public: resume QA ou evidence pack cible.
